# Web / 远程仿真优先策略（V0.1）

## 1. 背景

当前项目不假设工程师本机已安装 ngspice、LTspice、PLECS、PSIM 等 PC 端仿真工具。

因此，仿真架构需要优先支持以下两类能力：

1. 浏览器可访问的仿真工具
2. 智能体通过远程服务或连接器调用仿真能力

---

## 2. 核心原则

1. 浏览器工具可以作为入口，但不一定适合作为主证据引擎
2. 智能体应优先调用“远程 provider”，而不是依赖用户桌面环境
3. 电路级和系统级仿真应允许走不同 provider
4. provider 层要和模板层解耦，避免被某一厂商绑定

---

## 3. 推荐路线

### 3.1 电路级仿真

首选：

1. `remote_ngspice_service`
2. `agent_remote_sim_connector`

原因：

1. 更适合后端批量调度
2. 更适合参数扫描与自动判定
3. 更容易嵌入智能体工作流

浏览器备选：

1. `multisim_live_browser`
2. `easyeda_browser`

定位：

1. 快速概念验证
2. 轻量原理图级演示
3. 教学和协作预览

不建议将其作为电机控制器高风险问题的主验证平台。

### 3.2 系统级仿真

首选：

1. `simulink_online_browser`
2. `agent_remote_sim_connector`

原因：

1. 更适合控制算法和系统级工况分析
2. 更容易承载电机、逆变器、控制环、负载的联合模型

---

## 4. 智能体调用模式

建议支持三种模式：

1. `browser`
   - 智能体输出仿真步骤、参数和判据，工程师在浏览器工具中执行
2. `remote_service`
   - 智能体直接调用后端仿真服务，适合自动化闭环
3. `agent_tool`
   - 智能体通过连接器或工具层路由到不同仿真 provider

长期推荐以 `remote_service + agent_tool` 为主。

---

## 5. 为什么不建议只做纯 Web GUI 工具

对电机控制专家智能体而言，真正关键的是：

1. 模板复用
2. 参数扫描
3. 结果结构化
4. 自动判定
5. 审计与回归

很多纯浏览器 GUI 工具在这些方面并不是最佳选择，尤其是在：

1. 功率级细节验证
2. 工况批量扫描
3. 智能体自动触发执行
4. 结果归档与版本化

---

## 6. 项目当前落地方向

当前代码骨架已调整为：

1. 允许模板声明 `recommended_provider_ids`
2. 允许请求声明 `provider_id` 和 `execution_target`
3. 允许系统列出 provider 目录并按 provider 路由
4. 默认优先走远程 / agent 调用语义，而不是本地工具语义

---

## 7. 下一步实施建议

1. 先实现 `remote_ngspice_service` 适配器协议
2. 再实现一个 `agent_remote_sim_connector` 抽象层
3. 浏览器工具只作为人工交互备用入口
4. 系统级仿真优先接 Web 可访问的控制建模环境

---

## 8. 结论

如果当前没有 PC 端仿真软件，最合理的路线不是退回纯手工，而是把智能体架构切换成“Web 可访问 + 远程可调用”的模式。

对于这个项目，真正推荐的方向是：

1. 浏览器工具负责可访问性
2. 远程服务负责自动化执行
3. 智能体负责选择 provider、下发任务、解释结果

这样后续无论接入哪家平台，整体架构都不需要大改。
